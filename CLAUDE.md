# CLAUDE.md — Initiative Zero Prototype Enhancement Spec

## Context

This is a working Flask application demonstrating an AI-native code modernization pipeline for a Wealthsimple AI Builders application. The prototype makes real Claude API calls, uses SQLite for state, and has a single-page frontend. It already works end-to-end through 6 zones.

The goal of this enhancement pass is to make the prototype feel like a production internal tool, not a demo. Every change should reinforce the narrative that this system is real, opinionated, and built by someone who ships.

**Do not break existing functionality.** All current API routes, zone navigation, and Claude API integration must continue to work after changes.

---

## 1. Standardize Error Handling Across All Flask Routes

**Files:** `app.py`

Every API route must return errors in a consistent format:

```json
{"error": "Human-readable message", "error_code": "ZONE_2_ANALYSIS_FAILED", "zone": 2}
```

### Specific changes:

- Add a helper function at the top of `app.py`:

```python
def api_error(message: str, zone: int, code: str = "UNKNOWN", status: int = 500):
    return jsonify({"error": message, "error_code": code, "zone": zone}), status
```

- Wrap every route handler body in try/except. Currently `api_run_extraction`, `api_run_generation`, `api_coexistence_simulate`, and `api_production_decision` have inconsistent or missing error handling.

- Use the helper consistently. Map existing error returns to the new format. For example, in `api_run_analysis`:

```python
# Before
return jsonify({"error": "Source file not found"}), 404

# After
return api_error("Source file not found", zone=2, code="SOURCE_NOT_FOUND", status=404)
```

- The frontend (`app.js`) already reads `.error` from responses, so the new format is backward-compatible. No frontend changes needed for this.

**Files:** `external/tester.py`, `app.py`

- Rename the dict: `LEGACY_BEHAVIORS` → `KNOWN_TEST_VECTORS`

```python
# Known test vectors derived from legacy system execution traces.
# In production, these would be generated by running the actual legacy
# system against a controlled input corpus, not hardcoded.
# For this prototype, they represent verified legacy behavior.
KNOWN_TEST_VECTORS = { ... }
```

- Update all references in `tester.py` (the `run_tests` function and `build_test_harness` references)
- Update the one reference in `app.py` (`api_coexistence_simulate` imports `LEGACY_BEHAVIORS`)

```javascript
function toggleReasoning() {
  const body = document.getElementById('reasoning-body');
  const toggle = document.getElementById('reasoning-toggle');
  if (body.style.display === 'none') {
    body.style.display = 'block';
    toggle.classList.add('open');
  } else {
    body.style.display = 'none';
    toggle.classList.remove('open');
  }
}
```
